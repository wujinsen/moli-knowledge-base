## 一、范式转变：从存储容器到认知中枢

| 核心维度 | 传统知识库（2023 前） | 2026 代智能知识库 |
|----------|----------------------|-------------------|
| **交互模式** | 人找知识（关键词 / 向量检索） | **知识找人**（伴随式、情境感知推送） |
| **内容形态** | 静态图文（PDF、Word、Wiki） | **全模态流转**（音视频解析、3D、代码片段） |
| **执行深度** | 仅提供参考信息 | **Agent 化**：基于知识调用工具、交付可执行结果 |
| **价值度量** | 存了多少数据 | **能执行多少种 Action** |

**核心洞察**：知识库的价值不再取决于「存了多少」，而取决于 Agent 系统能基于这些数据执行多少种类的**行动（Action）**。

本项目起点是 [Karpathy LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)——已是「LLM 维护 wiki」而非纯静态 CMS；演进方向是在此基础上补齐 2026 五能力，而非推倒重来。

## 二、2026 五大核心能力（定义）

### 1. 原生多智能体协同（Agentic Orchestration）

知识库内置多重智能体（研究员、程序员、数据分析等）。面对复杂需求时：

- 自主拆解任务
- 调度不同 Agent 查资料、对比数据
- 交付可执行结果或直接调用外部 API

### 2. 动态自修复知识图谱（Self-Healing Graph RAG）

向量检索与**原生图数据库**融合：

- **持续注入**：新文档自动抽取实体与关系，写入图谱
- **关系推理**：跨文档、跨域多跳（例：A 项目风险 → 负责过 A 的员工 B → 在 C 项目的类似合规问题）

### 3. 泛模态原生解析（Omni-Modal Ingestion）

非文字载体与文本同级处理：

- 音视频：发言人分离、摘要、决议提取、时间点问答
- 空间 / 代码：Figma 组件关系、跨仓库函数演变追踪

### 4. 原子化重组与「知识找人」（Context-Aware Delivery）

- 组织单位从「篇」变为**原子**（概念、观点、数据片段）
- **情境感知**：嵌入 IDE、协同文档、CRM，侧边栏主动推送刚更新的相关片段
- **千人千面**：同一 query，不同角色看到不同粒度与视角

### 5. 细粒度权限与可信溯源（Granular Security & Lineage）

- **实体级权限**：图谱节点级 ACL；同一份合同，普通员工见脱敏版，CFO 见完整分析
- **100% 可溯源**：每段 AI 总结带原始锚点（录音 3:42、PDF 段落、回目原文）

## 三、本项目现状对照（五能力 × 落地度）

| 能力 | 已落地 | 部分具备 | 尚未建设 | 古籍域特殊说明 |
|------|--------|----------|----------|----------------|
| **① Agent 协同** | `/ingest` `/graph` `/query` `/dream` `/guard` `/lint` 六命令（[`AGENTS.md`](../chinese-classical-texts/apps/classical-novels/AGENTS.md)） | LLM 维护 wiki；`/query` 回填 topic | 多 Agent 编排器、工具调用 API、任务分解 UI | Agent 须受 Trust Guard 约束，不得改 raw |
| **② Graph RAG** | `relations` → `*.relations.json`；ECharts 力导向图；四维红楼图谱设计 | Pagefind 全文检索；chapter `summary` 作检索语料 | Neo4j / 向量+图混合检索；ingest 自动抽关系 | 推论边（影射、探佚）与事实边分层 |
| **③ 泛模态** | raw 含 scan 预留；2D/3D 大观园规划 | 120 回结构化 txt ingest | 影印本 OCR、说书音频、脂评批语层 | 版本对勘（甲戌/程高）需多版本并列 |
| **④ 知识找人** | 阅读器 + 搜索 + 关系图路由 | chapter 锚点、实体页 | 阅读器情境推送；读者/学者/教师千人千面 | 悬停名片、纵切专题（饮食等）即情境入口 |
| **⑤ 溯源与权限** | raw 只读；`source` / `edition`；Trust Guard；矛盾边 `contradicts` | topic 页「论据带出处」 | 实体级 ACL；点击跳原文高亮；录音级锚点 | 学术可信度是生命线，优先于功能炫技 |

**结论**：本项目在 **⑤ 溯源** 与 **① 单 Agent 工作流** 上基础最好；**② Graph RAG** 有 schema 与静态图，缺动态图库与混合检索；**③④** 随 P4/P5（地图、NPC 对话）逐步补齐。

## 四、五能力 → 工程映射

### 4.1 Agentic Orchestration → 「六命令 + 编排层」

当前 AGENTS.md 的六命令已是**专职 Agent 雏形**：

| Agent 角色 | 命令 | 行动（Action） |
|------------|------|----------------|
| 文献员 | `/ingest` | 读回目 → 更新实体/关系 → 写 log |
| 图谱员 | `/graph` | 重建 relations.json |
| 研究员 | `/query` | 多页综合 → 带出处作答 → 可选回填 topic |
| 质检员 | `/guard` | 事实边回 raw 校验 |
| 巩固员 | `/dream` | 别名合并、补链、weight 重算 |
| 巡检员 | `/lint` | 孤儿页、缺链、矛盾报告 |

**S1 演进（近期）**：增加轻量 **Orchestrator**——用户说「整理第 27 回并更新图谱」，自动串 `/ingest` → `/graph` → `/guard`，而非逐步人工触发。

**S2 演进（中期）**：按用户画像路由子 Agent：

- **读者 Agent**：伴读、路径查询、2D 地图导航
- **红学 Agent**：版本对勘、探佚假说卡（inference 分层）
- **教师 Agent**：因果链、自测题生成

**S3 演进（远期）**：可调用外部工具（导出报告、Neo4j 查询、对话 API），交付物不仅是 markdown 页。

```
用户复杂需求
     │
     ▼
┌─────────────┐
│ Orchestrator │  拆解任务、选 Agent、汇总结果
└──────┬──────┘
       ├── Ingest Agent ──→ content/*.md
       ├── Graph Agent  ──→ *.relations.json
       ├── Query Agent  ──→ topics/ + 对话
       └── Guard Agent  ──→ 校验报告 / 阻断幻觉边
```

### 4.2 Self-Healing Graph RAG → 「Markdown 真相层 + 图副库」

**原则**（见 [`红楼梦-知识图谱架构.md`](../chinese-classical-texts/docs/红楼梦-知识图谱架构.md)）：git 可审计的 `content/` + `raw/` 为主库；图数据库为**构建期导出或查询副库**，不替代真相层。

| 阶段 | 能力 | 交付 |
|------|------|------|
| **G0（现状）** | frontmatter `relations` → JSON → ECharts | ✅ |
| **G1** | ingest 时 LLM 提议新边 → guard 校验 → 写入 md | 半自动「自修复」 |
| **G2** | Pagefind + 向量（BGE-M3）+ 图多跳混合检索 | Graph RAG QA |
| **G3** | Neo4j 可选服务；跨实体路径（晴雯—芙蓉—黛玉） | 红学探索 API |
| **G4** | 新 ingest 自动 merge 冲突边、升级 weight | 全链路 self-healing |

**跨域推理示例（古籍域）**：

> 问：「茄鲞出现在哪些社交场合，与哪些人物地位相关？」

检索链：向量命中 `dish/茄鲞` → 图跳 `出现于→第41回` → `参与人物→贾母/凤姐` → topic 纵切「饮食—地位」→ 合成答案，每跳带 `source`。

### 4.3 Omni-Modal Ingestion → 「古籍多版本 + 沉浸模态」

| 模态 | 来源 | 项目路径 | 优先级 |
|------|------|----------|--------|
| 结构化文本 | 殆知阁 txt | `raw/daizhige/` | ✅ 已用 |
| 善本 / 影印 | PDF、LaTeX | `raw/editions/`、`raw/scan/` | P2 版本对勘 |
| 批语层 | 脂评本 | raw 待补 | P2 脂学 |
| 空间 | 大观园 2D/3D | `apps/daguanyuan/`（规划） | P5 |
| 音频 | 说书、讲座 | `raw/external/` | P4 可选 |

ingest 流水线扩展：`/ingest` 不限于「一回 txt」，增加 **modal adapter**（OCR 批语 → `annotation` 实体；扫描页 → 段落锚点）。

### 4.4 Context-Aware Delivery → 「阅读情境 + 画像路由」

**原子化**：实体页段落、回目 `summary`、topic 结论块、关系边 evidence——均可独立索引与推送。

| 情境 | 推送内容 | 功能 ID |
|------|----------|---------|
| 阅读第 27 回 | 葬花吟、黛玉判词、潇湘馆 | R-01~03 |
| 浏览关系图选中「晴雯」 | 影射链、撕扇事件、芙蓉女儿诔 | R-10, 维度④ |
| 学者打开版本对勘 | 程高 vs 甲戌异文、脂批 | H-10~11 |
| 教师备课「大观园」 | 地点列表、第 17 回题额、建筑考证链 | T-01~05 |

**千人千面**（与 [`产品功能规划.md`](../chinese-classical-texts/docs/产品功能规划.md) 对齐）：

- 普通读者 → SOP 式伴读、美食地图
- 史学家 → SNA、名物定量、引文导出
- 师生 → 因果链、自测、备课包

### 4.5 Granular Security & Lineage → 「Trust Guard 升级版」

古籍项目**公开阅读为主**，权限需求低于企业合同库，但**溯源要求更高**。

| 机制 | 现状 | 目标 |
|------|------|------|
| 事实 vs 推论 | `inference: true` 边（规划） | 图谱/UI 虚线区分 |
| 原文锚点 | chapter path + 回目 | 段落级高亮跳转 |
| 版本 lineage | `edition` + `source` | 多版本并列谱系图 |
| guard 双层 | form + content 校验 | ingest 自动 guard |
| 实体级 ACL | — | 多租户域扩展时再建（兄弟领域如保险） |

**100% 溯源 UI**：AI 回答 / topic 页每条论据 → `[第27回 · 葬花吟]` 可点击 → 阅读器定位段落（H-12, B-15）。

## 六、多领域工作区如何复用

`moli-knowledge-base` 设计为**同级兄弟领域**（古籍、保险、教育…），五能力在**工作区层**复用、在**领域层**定制：

```
moli-knowledge-base/
  docs/                          ← 本文：跨域演进标准
  chinese-classical-texts/       ← 域 1：古籍（Trust Guard、Graph、六命令）
  <future-domain>/               ← 域 2：各自 raw/wiki/apps + 域级 ACL
  packages/                      ← 远期：shared orchestrator、graph-export、pagefind-ui
```

| 组件 | 共享 | 域定制 |
|------|------|--------|
| Orchestrator 框架 | ✅ | Agent 技能包、工具列表 |
| Trust Guard 模式 | ✅ | 校验规则（古籍回 raw，合同回 PDF） |
| Graph export | ✅ | 实体类型与关系词表 |
| Pagefind / 检索 | ✅ | 归一化策略（文言文 vs 现代语） |
| 实体级 ACL | ✅ | 策略 per 域 |

## 七、刻意不做（现阶段）

避免把 2026 愿景做成「功能清单堆砌」：

1. **不**用 Neo4j 替代 markdown 真相层
2. **不**在尚无脂本 raw 前做花架版本对勘 UI
3. **不**为单用户古籍站过早建设 CRM/IDE 插件级「知识找人」——先在阅读器内做情境推送
4. **不**让 Agent 自动写 raw 或合并版本——违反学术铁律
5. **不**跳过 Guard 追求 ingest 速度——幻觉比慢更致命
