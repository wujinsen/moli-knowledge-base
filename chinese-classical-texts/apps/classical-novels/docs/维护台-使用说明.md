# 维护台 · 使用说明

> 面向维护者的操作手册。协议细节见 [`维护台-聊天协议.md`](维护台-聊天协议.md)；流程宪法见 [`AGENTS.md`](../AGENTS.md)。

---

## 1. 维护台是什么

**维护台（Studio）** 是站点内的知识库自我进化模块，把 AGENTS.md 里的 `/ingest`、`/query`、`/lint`、`/graph`、`/dream`、`/guard` 接到可点击的 UI 上：

- **聊天 + 变更卡片**：描述意图 → 收到 `PatchProposal` → 点「应用全部」写盘
- **批处理仪表盘**：只读体检、图谱预览、睡眠巩固、Trust Guard
- **回目摄取**：对照原文维护 frontmatter 与登场人物（**不改 chapters 正文**）
- **待办推送**：crosslinks 缺口、摄取抽样、低密度人物 → 一点开聊

当前为 **本地 dev 专用**：生产构建不含顶栏「维护台」入口，Orchestrator API 也不对公网开放。

## 启动（必做）

需要 **两个终端**，都在 `chinese-classical-texts/apps/classical-novels` 下操作。

### 配置 LLM（可选，推荐）

维护台默认 **auto**：有 API Key 走真实 LLM，否则回退 Mock 样例。

在 `apps/kb-orchestrator` 复制 `.env.example` 为 `.env`，或在启动前 export：

```bash
# OpenAI 或任意 OpenAI 兼容端点
set STUDIO_LLM_API_KEY=sk-...
set STUDIO_LLM_MODEL=gpt-4o-mini
# set STUDIO_LLM_BASE_URL=https://api.openai.com/v1   # 自建/代理时修改

# 强制模式
# set STUDIO_AGENT_MODE=llm    # 无 key 则报错
# set STUDIO_AGENT_MODE=mock   # 永远用样例 proposal
```

健康检查会返回 `agentMode` / `llmModel`；聊天区顶部显示 **LLM · model** 或 **Mock 模式**。

LLM 可用工具：`read_chapter` · `read_entity` · `read_item` · `read_chapter_ingest` · `propose_patch`（diff 写盘前校验，chapters 正文只读）。

### 终端 A — Orchestrator API

```bash
npm run studio:api
```

等价于在 `apps/kb-orchestrator` 启动 FastAPI（默认 `:8787`）。  
`NOVELS_ROOT` 默认指向本应用的 `src/content` 与 `scripts/`。

健康检查：

```bash
curl http://127.0.0.1:8787/api/studio/health
```

应返回 `"status": "ok"` 且 `"novelsRootExists": true`。

### 终端 B — Astro 站点

```bash
npm run dev
```

终端会打印 **Local** URL（常见 `http://localhost:4321`，也可能 4322–4326）。  
Astro 已将 `/api/studio/*` 代理到 `127.0.0.1:8787`（见 `astro.config.mjs`）。

若聊天区提示「Orchestrator 未启动」，先确认终端 A 在跑，且 dev 端口与浏览器一致。

## 4. 界面分区

维护台页顶有三个 Tab + 右上角 **待办**。

### 4.1 待办

- 首次打开约 **15–20 秒**（摄取抽样扫描）；同书 **5 分钟内** API 有缓存
- 来源：`crosslinks` 缺口 · `ingest` 抽样 · 低密度人物（红楼梦 dream 压平后后者常为空）
- 点击一条：
  - **信物 / 情节类** → 切到「实体工作区」并自动发送建议 prompt
  - **ingest_chapter** → 切到「摄取」并定位到该回

### 4.2 实体工作区

左侧人物列表，右侧 **聊天 + 变更卡片**。

适用：**单人物** 信物、服饰、关键情节、问答。

| 快捷按钮 | 对应 intent | 说明 |
|----------|-------------|------|
| 补信物 | `fix_key_item` | crosslinks、名物页、第 N 回论据 |
| 改服饰 | `fix_costume` | frontmatter `服饰[]` 与名物页对齐 |
| 体检 | `run_lint` | 只读，不生成卡片 |

上下文 chips 显示信物/服饰数量、建议回目链接、crosslinks 缺口。

**示例对话**（MVP mock 有固定样例）：

- 打开 **贾迎春** →「补信物链」→ 累丝金凤 crosslinks 卡片
- 打开 **林黛玉** →「改服饰」→ 宫制堆纱花字段调整卡片

### 4.3 摄取

左侧回目列表，右侧 **摄取清单 + 摄取助手**。

对应 AGENTS **`/ingest`**：

1. 读清单：登场人数、缺页、正文未列入 frontmatter 的人、原文摘录
2. 点 **摄取本回** 或输入自然语言
3. 收到 proposal → **应用全部**（仅改 frontmatter / 人物页 / 数据 JSON，**不改 chapters HTML 正文**）
4. 手动：`index.md` + `log.md`（postApply 的 `logEntry` 会追加一条）

**示例**：`/honglou/studio?chapter=73` → 摄取本回 → 补绣桔、王住儿媳妇到 frontmatter，更新司棋/迎春情节。

版本下拉（红楼梦）：程高本 / 脂砚斋本，影响读哪份 chapter 文件与 `readUrl`。

### 4.4 批处理

四个子命令，与 AGENTS 一一对应：

| Tab | AGENTS | 写盘？ | 说明 |
|-----|--------|--------|------|
| `/lint` | `/lint` | 否 | 结构问题 + 人物密度分带；可点人物跳转实体工作区 |
| `/dream` | `/dream` | 是（需确认） | **三书**；按 tier 预览后 POST 应用 |
| `/graph` | `/graph` | 是（需确认） | 重建 `src/data/<书>.relations.json` |
| `/guard` | `/guard` | 否 | first_appear / 情节 / 关系 / 交易校验 |

`/dream`、`/graph` 应用前会弹确认；应用后跑 `postApply.scripts`（如 `build_relations.py`、`trust_guard.py`）。

## 5. 变更卡片（PatchProposal）怎么用

1. 助手回复下方出现 **变更卡片**：标题、摘要、出处（第 N 回链接）、patch 列表
2. 展开可看 unified diff 摘要
3. **应用全部** — 写盘 + 执行 postApply + 追加 `<书>.log.md`
4. **丢弃** — 不写入
5. `guardPreview` 为 pass / warn / skipped（MVP mock 多为示例值）

预览不写盘：

```bash
# 在 classical-novels 目录
set STUDIO_DRY_RUN=1   # Windows
export STUDIO_DRY_RUN=1 # macOS/Linux
# 再点「应用」— Orchestrator 需重启后读取环境变量
```

## 6. 与 AGENTS 命令对照

| 你在 Cursor 里打 | 在维护台里做 |
|------------------|--------------|
| `/ingest` 第 N 回 | 摄取 Tab · 或阅读器「维护本回」 |
| `/query` … | 实体工作区聊天（问答；值得回填时再存 topics） |
| `/lint` | 批处理 → `/lint` |
| `/graph` | 批处理 → `/graph` → 确认应用 |
| `/dream` | 批处理 → `/dream`（honglou）→ 选 tier → 应用 |
| `/guard` | 批处理 → `/guard` |

**维护台不能替代的部分**：

- 大段 **ingest 人工判断**（哪些配角要建页）仍要你拍板
- **topics 回填** 需确认是否存盘（见 AGENTS 回填规则）
- **git commit** — Studio 只写工作区文件，提交仍用 git

## 7. 推荐工作流

### A. 修一个人物信物链

1. 待办或 `/lint` 发现 crosslinks 缺口 → 点待办 / 打开该人物
2. 「补信物」→ 审卡片 diff → 应用
3. 批处理 → `/guard` 确认 0 unverified
4. 浏览器刷新人物页 / 图鉴 chip

### B. 摄取一回

1. 阅读器读完第 N 回 → **维护本回**
2. 看清单里「正文未列入」与缺页
3. 摄取本回 → 应用 → 检查 `log.md`
4. 批处理 → `/graph`（若改了 `relations`）

### C. 批量巩固（红楼梦）

1. `/lint` 看密度分带
2. `/dream` 选推荐 tier → 预览 stuck 列表 → 确认应用
3. `/graph` + `/guard`

## 8. 限制与现状（2026-06）

| 项 | 状态 |
|----|------|
| LLM | **已接入**（OpenAI 兼容 API + tool loop）；无 Key 时自动 **Mock** |

| 待办 ingest 扫描 | 每 5 回抽样 + 关键回，非全书 120 回全扫 |
| SQLite / 会话持久化 | 内存 store；重启 Orchestrator 丢 session |

## 9. 故障排查

| 现象 | 处理 |
|------|------|
| 聊天一直 loading / 报 fetch 错 | 开 `npm run studio:api`；看 `:8787/health`；确认 `.env.development` 含 `PUBLIC_STUDIO_API=http://127.0.0.1:8787` 并**重启 dev** |
| 批处理 / 待办显示 **Not Found** | 多为前端未连上 API（用了 preview/生产页，或 proxy 未生效）；勿用 `npm run preview` 开维护台 |
| 应用失败 422 | diff 与当前文件不一致；本地改过文件需重新生成 proposal |
| 待办为空 | 可能确实无缺口；或 API 报错（看浏览器 Network） |
| dev 端口不是 4321 | 用终端里 **Local** 那一行 URL |
| 改 API 路由后不生效 | 重启 `studio:api`（uvicorn `--reload` 有时漏加载） |
