# 知识库组织方案

> 记录时间：2026-06-14
> 主题：多个知识库（不同领域）该如何组织——延续 Karpathy「LLM Wiki」模式的讨论。

## 背景

古籍内容已归入 **`chinese-classical-texts/`**（中国古籍领域）：

- `raw/daizhige/集藏/小说/` — 红楼梦、金瓶梅等章回小说全文
- `raw/editions/jinpingmei/` — 金瓶梅善本（JingPingMei-ZhCN、崇祯本、孟梅馆）
- `raw/editions/xiyouji/` — 西游记世德堂本
- `raw/daizhige/` — 殆知阁全集
- `apps/classical-novels/` — 章回叙事 wiki + 前端

问题：三个小说库放一起是否合适？若以后加入教育类、保险类等其它领域，能否放一起？

## 结论

- **三部小说放一起：可以，且推荐。** 共享同一套「人物 / 回目 / 主题」结构，可用一份 schema，甚至跨书交叉引用。
- **不同领域（教育、保险等）：不建议混在同一个 wiki 里，应按「领域」分顶层。**

判断标准一句话：**会不会互相引用？会 → 放一起；不会 → 分开。**

## 为什么不同领域要分开

LLM Wiki 的机制是「查询时先读 `index.md` 再深入」。混域会带来：

| 问题 | 后果 |
|---|---|
| schema 冲突 | 小说要「人物/回目」，保险要「产品/条款/理赔案例」，一份规范说不清 |
| index 被撑大 | 问保险却要先翻过几百条小说条目，检索变差、token 浪费 |
| 交叉引用失真 | 「西门庆」与「重疾险」之间没有有意义的链接，graph view 变噪音 |
| log / lint 混乱 | 巡检时不同域的矛盾混在一起，难维护 |

## 推荐结构（按领域分顶层）

```
moli_project/
  chinese-classical-texts/     # 领域1：中国古籍
    raw/  wiki/  apps/  scripts/
  insurance/                   # 领域2：保险（示例英文名）
    raw/  wiki/  apps/
  education/                   # 领域3：教育
    raw/  wiki/  apps/
```

要点：

- 每个领域是一个**自洽的 wiki 单元**——自己的 schema、`index.md`、`log.md`、lint 流程。
- 同域内部（如三部小说）可共享规范、互相链接。
- 跨领域彼此隔离，互不引用。
- **进阶**：若某领域以后要单独同步 / 分享 / 做 git 仓库，再升级为独立 repo。当前一个仓库分顶层目录即可。

## 对当前项目

`chinese-classical-texts/` 已作为古籍领域单元落地。章回叙事见 `apps/classical-novels/AGENTS.md`。

## 相关概念回顾（LLM Wiki 模式）

三层架构：

1. **Raw sources**：原文，只读，真相源（如各书 `ch/`）。
2. **The wiki**：LLM 生成维护的 markdown 页面（人物 / 主题 / 回目 / 综述）。
3. **The schema**：`AGENTS.md` / `CLAUDE.md`，规定结构、约定、工作流。

三个操作：**Ingest（摄取来源进 wiki）→ Query（提问，好答案回填）→ Lint（定期体检）**。
